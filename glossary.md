# 用語集

本書に登場した主な用語を、五十音・アルファベット順にまとめます。詳しい説明は各章を参照してください。

## あ行

**意味解析（semantic analysis）**
: 構文解析でできた AST を調べ、名前の解決や型の整合性など、形だけでは分からない「意味の正しさ」を確かめ、実行に必要な情報を補う工程。

**インタプリタ（interpreter）**
: プログラムを事前にまとめて機械語へ翻訳することなく、その場で読みながら実行する処理系。JIT を組み合わせた実装では実行中に機械語を生成することもある。本書では木をたどる方式（AST インタプリタ）とバイトコード方式（VM）を扱った。

**インラインキャッシュ（inline cache）**
: メソッドや演算の探索結果を、その呼び出し地点のそばに覚えておき、次回同じ型なら探索を省く高速化技法。

**インライン展開（inlining）**
: 関数呼び出しを、その関数の中身で置き換える最適化。呼び出しの手間を省き、他の最適化の機会も広げる。

**AOT コンパイル（Ahead-Of-Time compilation）**
: プログラムを実行する前に、まとめて機械語へ翻訳しておく方式。

**AST（抽象構文木, Abstract Syntax Tree）**
: プログラムの構造を表す木。構文解析の成果物で、優先順位や括弧は木の形に吸収される。

## か行

**ガベージコレクション（garbage collection, GC）**
: 使われなくなったヒープ上のメモリを、処理系が自動で見つけて回収する仕組み。

**環境（environment）**
: 「変数名 → 値」の対応表。関数呼び出しごとに新しく作られ、スコープを実現する。

**機械語（machine code）**
: CPU が直接実行できる命令。言語処理系の最終的な翻訳先のひとつ。

**クロージャ（closure）**
: 自分が定義された環境（外側の自由変数）を捕まえて持ち運ぶ関数。

**コルーチン（coroutine）**
: 途中で自分から実行を中断し、あとで続きから再開できる、軽量な並行の仕組み。

**コンパイラ（compiler）**
: あるプログラムを別の言語（機械語やバイトコードなど）へ翻訳する処理系。

**構文解析（parsing）**
: トークン列を文法に照らし、AST に組み上げる工程。

## さ行

**字句解析（lexical analysis, lexing）**
: 文字の並びを、意味のある最小単位であるトークンに区切る工程。

**自由変数（free variable）**
: ある関数の中で使われているが、その関数では定義されていない（外側から借りている）変数。クロージャが捕まえる対象。

**スコープ（scope）**
: 変数が見える（有効な）範囲。本書の MiniRuby は「関数ごとに閉じる」単純な規則を採用。

**スタック（stack）**
: 後入れ先出し（LIFO）でデータを積み下ろしする構造。VM の計算途中や、呼び出しフレームの管理に使う。

**スタックマシン（stack machine）**
: 計算の途中結果をスタックに置く方式の VM。命令が短く、コンパイラが書きやすい。

**スレッデッドコード（threaded code）**
: 中央の分岐（case 文）をなくし、各命令処理の末尾から次の命令処理へ直接飛ぶ、ディスパッチ高速化技法。

## た行

**タグ付き表現（tagged representation）**
: 値に「種類を表す札（タグ）」を添えて、整数・文字列などを区別する値表現。

**脱最適化（deoptimization）**
: JIT が仮定つきで生成した機械語の実行中に仮定が崩れたとき、実行状態をインタプリタに移し替えて安全に続行する仕組み。

**定数畳み込み（constant folding）**
: コンパイル時に答えの分かる計算を、その場で計算済みの値に置き換える最適化。

**デッドコード除去（dead code elimination, DCE）**
: 実行結果に影響しないコードを取り除く最適化。

## な行

**二返戻値法（two-return-value method）**
: 例外を使わず、「値」と「エラー標識」の 2 つを戻り値で返してエラーを伝える方式。Go の `value, err` が代表例。

## は行

**バイトコード（bytecode）**
: VM が実行する、単純な命令の列。AST をコンパイルして得る中間表現の一種。

**バックパッチ（backpatching）**
: ジャンプの飛び先が後で決まる場合に、いったん空けて命令を出し、確定後に正しい番地で埋め戻す手法。

**PC（program counter）**
: いま何番目の命令を実行しているかを指す番号。VM の実行位置を表す。

**ピープホール最適化（peephole optimization）**
: 数命令ぶんの小さな窓でコードを見て、無駄な並びを短い並びに置き換える局所的な最適化。

**フレーム（frame）**
: 関数呼び出しごとの実行情報（ローカル変数・PC など）をまとめたもの。呼び出しでスタックに積まれる。

**部分評価（partial evaluation）**
: 入力の一部が分かっているとき、その分を先に計算して特殊化したプログラムを作ること。Futamura 投影で「コンパイル」と結びつく。

**ホットスポット（hot spot）**
: プログラム中で特に頻繁に実行される部分。JIT はプロファイリングでここを見つけ、機械語化の対象にする。

## ま行

**マークアンドスイープ（mark and sweep）**
: ルートから到達できるオブジェクトに印をつけ（マーク）、印のないものを回収する（スイープ）GC 方式。

**命令融合（instruction fusion）**
: 頻出する命令の並びを 1 つの命令にまとめ、ディスパッチ回数を減らす高速化。スーパー命令とも。

**メソッドディスパッチ（method dispatch）**
: メソッド呼び出しのときに、相手の型・クラスから対応するメソッド本体を探す処理。

## ら行

**レキシカルスコープ（lexical scope, 静的スコープ）**
: 変数がどの定義を指すかを、プログラムの字面（書かれた場所）で決める規則。現代の多くの言語が採用。

**レジスタマシン（register machine）**
: 計算の途中結果を、番号のついた作業場所（レジスタ）に置く方式の VM。命令数は少ないが命令が大きい。

## アルファベット

**FFI（Foreign Function Interface, 外部関数インタフェース）**
: 別の言語（主に C）で書かれた既存ライブラリの関数を、つなぎコードをほぼ書かずに呼び出す仕組み。

**JIT コンパイル（Just-In-Time compilation）**
: プログラムを実行しながら、頻繁に実行される部分（ホットスポット）を機械語へ翻訳する方式。

**LICM（loop-invariant code motion, ループ不変式の巻き上げ）**
: ループ中で毎回同じ結果になる計算を、ループの外へ追い出す最適化。

**TCO（tail call optimization, 末尾呼び出し最適化）**
: 関数の最後でする呼び出しについて、今のフレームを使い回し、スタックを消費せずに実行する最適化。

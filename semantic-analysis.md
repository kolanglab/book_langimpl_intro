# 意味解析 ── 木に意味を通す

構文解析が終わると、プログラムの**形**を表す AST が手に入ります。しかし、形が整っていることと、**意味が通っている**ことは別問題です。`undefined_var + 1` は文法的には完璧な式ですが、`undefined_var` がどこにも定義されていなければ、実行のしようがありません。AST を調べて、こうした「形だけでは分からない誤り」を見つけ、実行に必要な情報を補う工程が **意味解析（semantic analysis）** です。

この章では、意味解析が処理系のどこに位置し、何をするのかを、流れの中で押さえます。本書は意味解析そのものには深入りしません ── その詳細は姉妹編に譲り、ここでは「構文解析と実装の間に、もう一段階ある」ことを理解するのが目標です。

## 構文解析と実装の「あいだ」

第I部で見た処理系の流れを、もう一度この章の視点で描き直してみましょう。

```mermaid
graph LR
    A["ソース"] --> B[構文解析]
    B --> C["AST"]
    C --> D[意味解析]
    D --> E["注釈付き AST"]
    E --> F["インタプリタ / VM / コード生成"]
```

意味解析は、構文解析（形を作る）と実装（実行する・コードを吐く）の**ちょうど中間**に位置します。前の工程からは「形は正しい AST」を受け取り、後の工程へは「意味も確かめられ、必要な情報が書き込まれた AST」を渡します。小さなインタプリタではこの工程を省いて、実行しながら誤りに気づく（＝実行時エラーにする）こともできます。実際、本書の基礎編のインタプリタも、独立した意味解析フェーズは持たせず、実行時にチェックします。しかし、規模が大きくなるほど「**実行する前に**まとめて誤りを見つけたい」という要求が強くなり、意味解析が独立した工程として重要になります。

## 意味解析が答える問い

意味解析が扱う代表的な問いを、MiniRuby を例に挙げてみましょう。

**名前は解決できるか（名前解決, name resolution）。** `puts total` の `total` は、どこかで定義された変数でしょうか。前章のスコープ規則に照らして、「この名前が、いまこの場所から見える変数や関数を指しているか」を突き止める作業です。見つからなければ「未定義の変数」エラーになります。関数呼び出し `add(1, 2)` についても、`add` という関数が定義されているか、引数の個数（ここでは 2 個）が定義と合っているかを確かめられます。

**型は整合しているか（型検査, type checking）。** MiniRuby は整数しか扱わないので型の問題はほとんど起きませんが、応用編で文字列や配列を入れると話が変わります。`"abc" * "def"` のような「文字列同士の掛け算」は意味をなしません。こうした「演算とデータ種類の食い違い」を実行前に検出するのが型検査です。静的型付き言語（C、Java、Rust など）では、これがコンパイル時の中心的な仕事になります。

**実行に必要な情報を補えるか（情報の付与）。** 意味解析の成果は「OK/NG の判定」だけではありません。たとえば「このローカル変数は、関数のフレームの何番目に置く」という割り当てを決めて AST に書き込んでおけば、後段の VM はその番号を使って高速に変数へアクセスできます。本書の VM の章では、まさにこの「変数に番号を振る」作業を行います ── これは小さいながらも立派な意味解析の一種です。

> [!NOTE]
> 意味解析の結果として、AST の各ノードに「型は Integer」「格納先は 2 番」といった**注釈（annotation）** を書き加えた木を、**装飾構文木（decorated AST）** と呼びます。後の工程は、生の AST ではなくこの装飾済みの木を入力にします。

## なぜ独立した工程にするのか

意味解析を「実行時のチェック」で済ませず、わざわざ独立した工程として前倒しする利点は何でしょうか。

最大の利点は **誤りを早く、まとめて見つけられる**ことです。実行時チェックだと、エラーはそのコードが実際に通過したときにしか出ません。めったに通らない分岐のタイプミスは、運が悪いと本番で初めて発覚します。一方、意味解析で事前に木全体を走査すれば、実行する前にすべての未定義変数や型の不整合を洗い出せます。バグの発見は早ければ早いほど安く済む、というのは処理系設計でも変わりません。

もうひとつの利点は **後段を単純にできる**ことです。「ここまでに名前解決と型検査は済んでいる」と保証されていれば、コード生成器は「変数が存在するか」「型が合うか」を気にせず、変換だけに集中できます。役割を分けることで、各工程がすっきりするのです[Aho et al., 2006](#cite:aho2006)。

## 本書での扱い

意味解析は、それだけで一冊の本になるほど奥が深い分野です。名前解決のスコープ規則、型推論、データフロー解析、抽象解釈[Cousot and Cousot, 1977](#cite:cousot1977) ── こうした話題は、姉妹編『[言語処理系の意味解析入門](https://kolanglab.github.io/book_semantic_intro/#cover)』で詳しく扱っています。プログラムを実行せずにその性質を見抜く技術に興味がわいたら、ぜひそちらへ進んでください。

本書では、意味解析を独立した大きな章にはせず、必要なところで最小限のチェックや情報付与を行う、という現実的な方針をとります。具体的には ──

- **AST インタプリタ**（次章）では、変数や関数が見つからない場合は実行時に素直にエラーを出します。
- **仮想マシン**（次々章）では、コンパイル時に「ローカル変数に番号を振る」という軽い意味解析を行います。

つまり本書では、意味解析を「重い検証フェーズ」としてではなく、「**実装を成り立たせるために必要なだけ行う下ごしらえ**」として登場させます。それでも、構文解析と実装の間にこの工程が確かに存在し、規模が大きくなれば独立させる価値があることは、頭に留めておいてください。

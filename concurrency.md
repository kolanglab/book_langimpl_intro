# 並行制御

ここまでの MiniRuby は、命令を上から順に、一本道で実行してきました。しかし現実のプログラムは、複数の処理を「同時に」進めたいことがよくあります。ファイルを読みながら画面を更新する、たくさんの通信を並行してさばく、複数の CPU コアをフルに使って計算を速くする ── こうした要求に応えるのが **並行制御（concurrency control）** です。この章では、並行処理を支えるいくつかの仕組み ── コルーチン、スレッド、プロセス、async/await ── を、処理系の視点で整理します。

## 並行と並列 ── まず言葉を分ける

最初に、混同しやすい 2 つの言葉をはっきり区別しておきます。

- **並行（concurrency）**：複数の処理を「**進行中**」にすること。必ずしも同時に動いている必要はなく、少しずつ交代で進めても構いません。一人の料理人が、煮込みの合間に野菜を切り、また鍋を見る ── これが並行です。
- **並列（parallelism）**：複数の処理を「**物理的に同時**」に動かすこと。複数の CPU コアで本当に同時に計算するのが並列です。料理人が複数人いて、それぞれ別の料理を同時に作る ── これが並列です。

```mermaid
graph TD
    subgraph concurrent["並行（交代で進める）"]
        A["タスク1 → タスク2 → タスク1 → ..."]
    end
    subgraph parallel["並列（同時に進める）"]
        B["タスク1（コア1）"]
        C["タスク2（コア2）"]
    end
```

並行は「複数の仕事をうまく**やりくり**する」話、並列は「複数の計算資源で**速くする**」話です。両者は重なることもありますが、目的が違います。並行は I/O 待ちの多いプログラム（サーバなど）で効き、並列は計算の重いプログラムで効きます。

## 仕組みを軽い順に並べる

並行・並列を実現する仕組みは、いくつかの段階があります。「どれだけ独立しているか」「切り替えのコスト」で軽い順に並べてみましょう。

```mermaid
graph LR
    A["コルーチン<br/>（最も軽い）"] --> B["スレッド"] --> C["プロセス<br/>（最も重い・独立）"]
```

### コルーチン ── 自分から譲る

**コルーチン（coroutine）** は、最も軽い並行の仕組みです。ふつうの関数は、一度呼ばれたら最後まで走って戻ります。コルーチンは、途中で**自分から実行を中断（yield, 譲る）** し、あとでその続きから再開できる関数です。中断しても、ローカル変数や実行位置はそのまま保たれます。

ここで、基礎編で作った VM の知識が効いてきます。コルーチンの「中断して、あとで続きから再開する」を実現するには、**そのコルーチンの実行状態 ── スタックとフレームと PC ── をまるごと保存しておけばよい**のです。基礎編の VM は、PC・スタック・フレームを明示的なデータとして持っていました。だから、それらを一式どこかに退避しておき、再開時に書き戻せば、続きから実行できます。

```ruby
# コルーチンの状態 = VM の実行状態をまるごと保存したもの
Coroutine = Struct.new(:stack, :frames, :status)  # status: :ready / :suspended / :dead
```

コルーチンの切り替えは、関数呼び出しと違って OS をまったく介さず、処理系の中だけで完結します。だから非常に軽く、何万個でも作れます。「誰がいつ実行を譲るか」をプログラム自身が決める（協調的, cooperative）のが特徴で、譲るタイミングが明示的なので、後述する競合の問題も起きにくいです。コルーチンは、歴史的にもコンパイラの実装技法として古くから知られてきました[Conway, 1963](#cite:conway1963)。

### スレッド ── OS が切り替える

**スレッド（thread）** は、ひとつのプログラム（プロセス）の中で、複数の実行の流れを持つ仕組みです。コルーチンと違い、**いつ切り替えるかを OS が決めます**（プリエンプティブ, preemptive）。プログラムが譲らなくても、OS が適当なタイミングで「はい交代」と別のスレッドに切り替えます。複数の CPU コアにスレッドを割り当てれば、**本当の並列**が実現できます。

スレッドはメモリ（ヒープ）を共有するため、データを受け渡しやすい反面、**競合状態（race condition）** という厄介な問題を抱えます。2 つのスレッドが同じ変数を同時に書き換えようとすると、結果が壊れることがあるのです。前章までに作った「即値でない値」の更新や、GC のさなかにスレッドが動くと、処理系自体が壊れかねません。これを防ぐため、ロック（lock, 排他制御）などの仕組みが必要になります。

> [!NOTE]
> Ruby（CRuby）や Python（CPython）には **GIL（Global Interpreter Lock, グローバルインタプリタロック）** という仕組みがあり、「一度に 1 つのスレッドしか処理系内部を実行できない」ように制限しています。これは処理系の内部構造（前述の値表現や GC）をスレッド競合から守るための割り切りで、おかげで処理系の実装は単純になりますが、計算が重い処理では複数コアを活かせなくなります。GIL を持つか持たないかは、処理系設計の大きな分岐点です。

### プロセス ── 完全に独立させる

**プロセス（process）** は、メモリ空間そのものを分けて、完全に独立した実行単位にする方式です。プロセス同士はメモリを共有しないので、競合状態は起きません。その代わり、データをやりとりするには、プロセス間通信（メッセージの送受信など）という明示的な手段が必要で、切り替えや生成のコストも最も重くなります。「メモリを共有しない」という割り切りで安全を買う方式で、Erlang のアクターモデルや、Ruby の `Ractor` などがこの発想に近いです。

## async/await ── 軽量並行の書きやすい形

I/O 待ちの多いプログラム（Web サーバなど）では、「待っている間に別の仕事を進める」並行が効きます。これをコルーチンで書くこともできますが、近年は **async/await** という構文を備える言語が増えました（JavaScript、Python、Rust など）。

`await` は、「ここで I/O などの結果を待つが、待っている間は処理系に制御を返して、別の仕事を進めてよい」という印です。`async` 関数は、内部的にはコルーチンのように「中断・再開できる関数」へと変換されます。処理系（やランタイム）は、たくさんの async 関数を **イベントループ** で管理し、「I/O が終わったものから順に再開する」というやりくりをします。見た目はふつうの逐次プログラムに近いのに、裏では大量の処理が交代で進む ── これが async/await の狙いです。実装の核心は、結局のところ「コルーチンによる中断・再開」であり、基礎編で作った VM の状態保存の応用線上にあります。

## 並列処理という別の難しさ

ここまでは主に「やりくり」としての並行を見てきました。複数の CPU コアで本当に同時に計算する **並列処理** には、また別の難しさが加わります。データをどう分割するか、コア間でどう同期するか、メモリの一貫性をどう保つか ── これらは処理系設計の最先端の話題であり、GIL の有無、メモリモデル、ロックフリーなデータ構造など、深く広い世界が広がっています。

並行・並列は、それ自体で一冊の本になるテーマです。スレッド・コルーチン・並列実行の詳細な仕組みと、処理系がそれをどう支えるかに踏み込みたい読者は、姉妹編『[並行・並列言語処理系](https://kolanglab.github.io/book_parallel_lang/#cover)』を参照してください。

> [!IMPORTANT]
> 並行制御の各方式は、「**安全さ・性能・書きやすさ**」のトレードオフで選ばれます。プロセスは安全だが重く、スレッドは速いが競合に弱く、コルーチンは軽いが自分で譲る必要がある。どれかひとつが万能なのではなく、解きたい問題に応じて使い分ける ── これが並行プログラミングの勘所であり、処理系はそれらの選択肢を支える土台を提供します。
